This tutorial puts you through the full BattleChain cycle — from both ends. You'll deploy a deliberately vulnerable contract as a protocol, set up its Safe Harbor agreement, and open it for attack. Then you'll switch sides and drain it as a whitehat.

Same wallet. Same chain. Two perspectives.

By the end you'll understand CEI violations, reentrancy exploits, and how Safe Harbor makes this whole thing legal and structured.

## Part 1 — Deploy the Vault

The `VulnerableVault` contract accepts token deposits and lets users withdraw their full balance. It's clean, simple, and broken in exactly one way.

Open `src/VulnerableVault.sol` and look at `withdrawAll()`:

```solidity
function withdrawAll() external {
    uint256 amount = balances[msg.sender];
    require(amount > 0, "VulnerableVault: nothing to withdraw");

    // ❌ INTERACTION before EFFECT
    token.transfer(msg.sender, amount); // external call — may trigger a callback

    // ❌ Effect happens here — but re-entrant calls already passed the check above
    balances[msg.sender] = 0;
}
```

The bug is subtle: the token transfer fires **before** the balance is cleared. If the token notifies the recipient on transfer — which ours does — an attacker can call `withdrawAll()` again inside that notification, before the balance ever hits zero.

This is the **Checks-Effects-Interactions** (CEI) violation. The correct version clears the balance first:

```solidity
// ✅ Correct CEI order
balances[msg.sender] = 0;          // Effect first
token.transfer(msg.sender, amount); // Interaction second
```

You've shipped the wrong one. Now deploy it.

<Tip>
Make sure you have enough demo tokens in your wallet to seed the vault (the script transfers `1,000` tokens from your address on deployment).
</Tip>

```bash
forge script script/Setup.s.sol --rpc-url battlechain --broadcast
```

Expected output:

```

Vault seeded with 1000 tokens

--- Add to your .env ---
VAULT_ADDRESS=0x...
```

Copy `VAULT_ADDRESS` into your `.env`.

## Part 2 — Create a Safe Harbor Agreement

The Safe Harbor agreement is the legal contract between you and anyone who attacks your vault. It defines the bounty terms and which contracts are in scope.

```bash
forge script script/CreateAgreement.s.sol --rpc-url battlechain --broadcast
```

This script:

1. Creates an agreement with a **10% bounty**, retainable from recovered funds
2. Puts `VAULT_ADDRESS` in scope on BattleChain (`eip155:627`)
3. Sets your wallet as the asset recovery address — stolen funds come back here
4. Locks the terms for **30 days** so they can't be changed mid-attack

Expected output:

```
Agreement created: 0x...
Commitment window extended 30 days
Safe Harbor adopted

--- Add to your .env ---
AGREEMENT_ADDRESS=0x...
```

Copy `AGREEMENT_ADDRESS` into your `.env`. Also set `RECOVERY_ADDRESS` to the wallet address you're deploying from.

## Part 3 — Request Attack Mode

Submit the vault for DAO review. Once approved, it's open season.

```bash
forge script script/RequestAttackMode.s.sol --rpc-url battlechain --broadcast
```

Your agreement is now in `ATTACK_REQUESTED` state. The DAO will verify:
- The vault is a new deployment, not a mainnet copy
- The bounty terms are reasonable
- The scope is clearly defined

To check approval status:

```bash
cast call 0x9E62988ccA776ff6613Fa68D34c9AB5431Ce57e1 \
  "getAgreementState(address)(uint8)" \
  $AGREEMENT_ADDRESS \
  --rpc-url http://testnet.battlechain.com:3051
```

| Value | Meaning |
|-------|---------|
| `2` | `ATTACK_REQUESTED` — awaiting DAO approval |
| `3` | `UNDER_ATTACK` — approved, vault is attackable |

Wait until you see `3` before continuing.

<Note>
Approval is manual. For the testnet tutorial, it typically happens within a few hours. If you're blocked, reach out in [Discord](https://discord.gg/cyfrin).
</Note>

## Part 4 — Understand the Exploit

Before running anything, understand exactly what's about to happen. Open `src/Attacker.sol`.

The attack exploits the token's transfer callback. When the vault calls `token.transfer(attacker, amount)`, the token fires `onTokenReceived` on the recipient — your `Attacker` contract. At that moment, the vault's `balances[attacker]` hasn't been cleared yet. So you call `withdrawAll()` again. And again. Until the vault is empty.

The reentrancy chain looks like this:

```
attack()
└── vault.withdrawAll()
    └── token.transfer() → onTokenReceived()
        └── vault.withdrawAll()          ← balance still non-zero
            └── token.transfer() → onTokenReceived()
                └── vault.withdrawAll()  ← balance still non-zero
                    └── ...              ← repeats until vault is empty
```

Once the vault is drained, the `_attacking` flag stops the loop. Then comes the Safe Harbor settlement:

```solidity
uint256 total = token.balanceOf(address(this));
uint256 bounty = (total * bountyBps) / 10_000; // 10%

token.transfer(recoveryAddress, total - bounty); // return 90% to protocol
token.transfer(owner, bounty);                   // keep 10%
```

You're not stealing — you're collecting. The protocol gets most of their funds back. You keep the agreed bounty. Everyone knew the rules.

Step back and look at the full picture.

As the **protocol**, you deployed a contract with a real vulnerability and put real tokens at risk on a live chain. The DAO reviewed your terms and opened your vault to the world. You didn't know when or how it would be hit.

As the **whitehat**, you found the vulnerability, built an exploit, and executed it cleanly within the Safe Harbor framework. You returned the majority of funds and kept your earned bounty.

This is exactly the dynamic BattleChain is designed to create — real economic pressure, real exploits, real consequences, without the real disaster of it happening on mainnet.
